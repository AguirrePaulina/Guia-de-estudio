Ingeniería de Software
Qué es el software · SDLC · Modelos de proceso · Roles
▾
Software: conjunto de programas, datos e instrucciones que permiten a una computadora realizar tareas. Es intangible y puede copiarse, distribuirse y modificarse.

Características de un BUEN software

Funcionalidad

Realiza las tareas para las que fue diseñado.

Confiabilidad

Funciona correctamente bajo condiciones normales.

Eficiencia

Usa recursos de forma óptima.

Mantenibilidad

Puede ser modificado y mejorado fácilmente.

Usabilidad

Es simple, intuitivo y fácil de entender.

SDLC — Ciclo de vida del software

Proceso estructurado con 6 fases: Requerimientos → Diseño → Implementación → Pruebas → Despliegue → Mantenimiento.

⚠️ "Sistemas legacy" NO es una fase del SDLC. Las fases son las 6 mencionadas.

Modelos de proceso

ModeloDescripciónCuándo usar
CascadaLineal y secuencial. Cada fase debe completarse antes de pasar a la siguiente.Requerimientos estables y bien definidos.
Iterativo/IncrementalCiclos cortos que agregan funcionalidad gradualmente.Proyectos con cambios frecuentes.
EspiralCombina cascada con prototipos y análisis de riesgo.Proyectos grandes y complejos.
PrototipadoImplementación parcial rápida para validar requerimientos.Requerimientos cambiantes o poco claros.
ÁgilIterativo, colaborativo, entregas continuas de valor.Software con evolución continua.

Cohesión y Acoplamiento

Cohesión (alta = bueno)

Grado en que los elementos de un módulo están relacionados entre sí. Alta cohesión = el módulo hace UNA cosa bien.

Acoplamiento (bajo = bueno)

Grado de dependencia entre módulos. Bajo acoplamiento = cambios en uno no afectan a otros.

Caja Blanca vs Caja Negra

Caja Negra (Black-Box)

Se centra en entradas y salidas. No se conoce el código interno. Solo se valida el comportamiento desde la perspectiva del usuario.

Caja Blanca (White-Box)

Requiere conocer el código fuente. Se analiza el flujo de datos y estructuras lógicas. Común en pruebas unitarias.

Caja Gris (Gray-Box): Acceso limitado a la arquitectura. Enfoque híbrido.

TDD — Test Driven Development

Enfoque donde se escribe el test ANTES de escribir el código.
Ciclo: 🔴 RED (test falla) → 🟢 GREEN (código mínimo para pasar) → 🔵 REFACTOR (mejorar)

Ventajas: cobertura garantizada, mejor diseño, detecta casos borde.
Desventajas: lento al inicio, difícil en sistemas legacy.

COCOMO — Estimación de costos

MM = a × KLOC^b (Esfuerzo en meses-persona)

KLOC = Miles de líneas de código

Métricas: KPIs vs OKRs

KPIs (Key Performance Indicators)

Indicadores que miden el rendimiento actual de un proceso o proyecto. Son cuantificables. Ej: velocidad del equipo, % de bugs.

OKRs (Objectives & Key Results)

Marco que combina un objetivo cualitativo con resultados clave medibles para alcanzarlo. Más estratégico y orientado a metas.
💡 OKRs definen objetivos y cómo medirlos. KPIs hacen seguimiento del rendimiento de esos objetivos.

Gestión de riesgos — 4 pasos

1. Identificación → 2. Análisis (probabilidad e impacto) → 3. Planeación (mitigación) → 4. Monitorización (seguimiento continuo)

Pregunta 1
¿Qué es un Software y qué debe cumplir para ser un BUEN SW?

Software es el conjunto de programas, datos e instrucciones que permiten a una computadora realizar tareas. A diferencia del hardware, el software es intangible y puede copiarse, distribuirse y modificarse.

Para ser un buen software debe cumplir con:

  • Funcionalidad: realiza las tareas para las que fue diseñado.
  • Confiabilidad: funciona correctamente bajo condiciones normales.
  • Eficiencia: usa los recursos (CPU, memoria, etc.) de forma óptima.
  • Mantenibilidad: puede ser modificado y mejorado fácilmente.
  • Usabilidad: es intuitivo y fácil de usar por el usuario final.
Pregunta 2
Verdadero o Falso: Indicar cuál NO pertenece a una fase del Ciclo de Vida de Desarrollo del SW. Justificar.

Las opciones eran: Análisis: Gestión de Requerimientos / Diseño: Arquitectura / Implementación: Desarrollo / Sistemas legacy: Heredados / Pruebas: Testing / Mantenimiento.

La respuesta correcta es: FALSO — "Sistemas legacy: Heredados" NO es una fase del SDLC.

Justificación: Las 6 fases del ciclo de vida del software son Requerimientos, Diseño, Implementación, Pruebas, Despliegue y Mantenimiento. Los "sistemas legacy" o heredados son sistemas existentes de una organización, no una fase del proceso de desarrollo. El resto de las opciones sí corresponden a fases reales del SDLC.

Pregunta 3
Defina en qué se centran LEAN y XP.

LEAN se centra en eliminar el desperdicio (waste) en todos los procesos. Su objetivo es maximizar el valor entregado al cliente minimizando actividades que no agregan valor, optimizando el flujo de trabajo y fomentando la mejora continua. Prioriza la entrega rápida y aprender de la experiencia.

XP (Extreme Programming) se centra en la calidad técnica del código y la comunicación estrecha con el cliente. Sus prácticas clave son: Pair Programming (programación en pares), TDD (desarrollo guiado por tests), refactoring continuo, integración continua (CI) y releases pequeños y frecuentes. Es la metodología ágil con mayor énfasis en la excelencia técnica.

Pregunta 4
¿Qué son los requerimientos? Nombrar ejemplos de requerimientos NO Funcionales.

Los requerimientos son las necesidades y expectativas que debe cumplir un sistema de software para ser considerado exitoso. Describen lo que el sistema debe hacer (funcionales) o cómo debe comportarse (no funcionales).

Ejemplos de requerimientos NO Funcionales:

  • Rendimiento: "El sistema debe responder en menos de 2 segundos."
  • Seguridad: "Los datos del usuario deben estar cifrados con AES-256."
  • Usabilidad: "Un usuario nuevo debe poder completar el registro en menos de 3 minutos."
  • Escalabilidad: "El sistema debe soportar 10.000 usuarios concurrentes."
  • Disponibilidad (uptime): "El sistema debe estar disponible el 99.9% del tiempo."
  • Portabilidad: "La aplicación debe funcionar en iOS y Android."
  • Compatibilidad: "Debe ser compatible con navegadores Chrome, Firefox y Safari."
Pregunta 5
¿A qué nos referimos cuando decimos lenguaje UML? Dar ejemplos.

UML (Unified Modeling Language) es un lenguaje de modelado visual, NO un lenguaje de programación. Se usa para especificar, construir y documentar los artefactos de un sistema de software. Permite capturar decisiones y conocimiento sobre los sistemas que se construyen, comunicando la estructura y el comportamiento del sistema a todos los involucrados.

Ejemplos de diagramas UML:

  • Diagrama de clases: muestra las clases, sus atributos, métodos y relaciones (herencia, composición, asociación).
  • Diagrama de casos de uso: muestra los actores (usuarios, sistemas externos) y las funcionalidades que el sistema ofrece.
  • Diagrama de secuencia: muestra cómo interactúan los objetos en el tiempo.
  • Diagrama de estados: modela los estados posibles de un objeto y las transiciones entre ellos.
  • Diagrama de actividad: representa flujos de trabajo o procesos.
Pregunta 6
Dado los conocimientos obtenidos, ¿se pueden utilizar diferentes paradigmas de programación en un software? Justificar.

Sí, es totalmente posible y muy común en el desarrollo real.

Un software moderno puede combinar múltiples paradigmas de programación según las necesidades de cada parte del sistema. Esto se denomina programación multiparadigma.

Justificación: Los lenguajes modernos como Python, JavaScript, Kotlin o Scala soportan múltiples paradigmas. Por ejemplo:

  • Una aplicación puede usar OOP para modelar las entidades de negocio (clases de Usuario, Pedido, etc.).
  • Al mismo tiempo usar programación funcional para el procesamiento de datos (maps, filters, reducers).
  • Y usar SQL (declarativo) para las consultas a la base de datos.

Utilizar el paradigma adecuado para cada problema mejora la legibilidad, mantenibilidad y eficiencia del código. Lo importante es la consistencia y el criterio del equipo.

Pregunta 7
¿A qué nos referimos cuando hablamos de cohesión y acoplamiento?

Cohesión se refiere al grado en que los elementos dentro de un módulo o clase están relacionados entre sí y trabajan hacia un único propósito. Alta cohesión es deseable: significa que el módulo hace una sola cosa bien hecha (principio SRP de SOLID). Un módulo con baja cohesión mezcla responsabilidades no relacionadas, lo que lo hace difícil de mantener.

Acoplamiento se refiere al grado de dependencia entre módulos o clases. Bajo acoplamiento es deseable: significa que los módulos son independientes y los cambios en uno no afectan a otros. Un sistema fuertemente acoplado es frágil porque un cambio en un módulo puede romper otros.

Regla de oro: buscar alta cohesión + bajo acoplamiento para lograr un sistema mantenible y flexible.

Pregunta 10
¿Qué es una herramienta CASE y cuándo se implementa? Dar ejemplos.

CASE (Computer-Aided Software Engineering) son herramientas que automatizan o asisten en las distintas etapas del proceso de desarrollo de software, mejorando la productividad y la calidad.

¿Cuándo se implementa? A lo largo de todo el SDLC, desde la fase de análisis hasta el mantenimiento.

Ejemplos por categoría:

  • Modelado y diseño: UMLet, StarUML, draw.io (diagramas UML)
  • Generación de código: GitHub Copilot, Swagger/JHipster
  • Análisis de calidad: SonarQube, ESLint, Checkstyle
  • Gestión de proyectos: Jira, Trello, Azure DevOps
  • Control de versiones: Git + GitHub/GitLab
  • CI/CD y despliegue: Jenkins, GitHub Actions, Docker
  • Testing: Selenium, Postman, Appium
